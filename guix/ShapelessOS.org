* Introduction

ShapelessOS (formerly LiterativeDeclarationOS) is a Linux-based operating system where *documentation and code are inseparable*. Built on Linux From Scratch methodology, enhanced with GNU Guix package management, and orchestrated through Emacs org-mode, this system embodies a simple principle:

#+BEGIN_QUOTE
*If there is no documentation, there is no code.*
#+END_QUOTE

Every package, every configuration, every decision exists as extensively documented org-mode files. The system doesn't just have documentation—the documentation *is* the system definition.

* What Makes ShapelessOS Different

** Documentation as Source Code
Your system definition lives in org-mode files:
- Full explanations of what each component does
- Why it exists and what problems it solves  
- How it works, down to assembly/machine code level
- Historical context and design evolution
- Source code analysis even for binary packages

** Knowledge Graph Architecture
Files are interconnected nodes, not hierarchical directories:
- Navigate concepts via org-roam backlinks
- Tag and categorize with supertags
- Traverse dependency chains
- Understand relationships between components
- One file per concept, many links to it

** Immutable/Atomic Foundation
- Read-only root filesystem (SquashFS + OSTree + OverlayFS)
- Locked after build—no runtime modifications
- Updates create new system images
- Instant rollback to any previous state
- Zero-trust model: containers for everything except core services

** Bleeding-Edge via Git
- No version numbers, track git repositories directly
- Always up-to-date with upstream
- Automatic hash computation and updates
- Hourly update checks via lightweight script
- Fail-safe: keep old hash if upstream breaks

** Hybrid Reproducibility
- Guix builds packages reproducibly to /gnu/store
- GNU Stow creates traditional FHS symlinks to /usr
- Containers see normal paths, inherit from immutable base
- Best of both worlds: Guix power + traditional compatibility

* Architecture Overview

** Layer 0: Immutable Root (Core)
#+BEGIN_SRC
Custom kernel
bash (system compatibility)
nushell (modern interactive shell)  
scroll (Wayland window manager)
Docker engine
Headscale client (VPN mesh networking)
GNU Stow (symlink management)
#+END_SRC

All installed to /gnu/store, stowed to /usr, frozen read-only after build.

** Layer 1: Infrastructure Services
Running on root system (not containerized):
- Headscale server (mesh network control plane)
- Passbolt server (password/secret management)
- Atuin server (shell history sync)
- K3s control plane container (orchestrates workloads)

** Layer 2: Workload Containers
All user applications run containerized, managed by K3s:
- Web development environments
- Pentesting tools (offense and defense)
- Databases and services
- Custom applications

Each container based on same ShapelessOS root image.

* Core Technologies

** Guix Package Management
- Reproducible builds
- Binary substitutes (where available)
- Functional package management
- Rollback capability
- We use Guix as build backend, not its DSL

** MAAK (org-mode → Guix Bridge)
- Reads org-mode package declarations
- Fetches git sources, computes hashes
- Generates Guix package definitions
- Orchestrates builds and installations
- Updates org-mode files with computed hashes
- Integrates with Emacs for smooth workflow

** Emacs Integration
- Custom Emacs package (ldos-mode.el) included
- Keybindings for package management
- Automatic hash computation (C-c C-h)
- Build packages from org blocks (C-c C-b)
- Navigate knowledge graph via org-roam
- Your custom Emacs distro comes preconfigured

** SquashFS + OSTree + OverlayFS
- SquashFS: Compressed immutable base
- OSTree: Version management and atomic updates
- OverlayFS: Configuration overlays
- Combined: Secure, efficient, rollback-capable

** GNU Stow
- Manages symlinks from /gnu/store to /usr
- Containers see traditional FHS layout
- Coexistence of Guix + normal paths
- Declared in org-mode, automated by MAAK

* Package Management Philosophy

** Three Categories of Packages

*** Stable from Guix Upstream
Core system components that rarely need updates:
- glibc, gcc, binutils
- Kernel (when not custom building)
- bash, coreutils

Use Guix's tested, stable versions.

*** Bleeding Edge from Git
Development tools tracking upstream HEAD:
- neovim, helix, zellij
- rust-analyzer, typescript servers
- Modern CLI tools

Rebuilt automatically, updated hourly.

*** Custom ShapelessOS Packages
Our own code and configurations:
- ldos-tools, ldos-init
- Custom system utilities
- Specialization-specific software

*** Package Definition in Org-Mode
#+BEGIN_SRC org
* Package: helix-editor
** Why It Exists
Modern modal editor written in Rust. Better treesitter integration
than neovim, native LSP support, simpler configuration.

** Design Philosophy

explanation of selection-first editing model]

(package-definition
  (name "helix")

  (build-system 'cargo)
  (stow-target "/usr/local"))
#+END_SRC
#+END_SRC

** Automatic Updates
- Lightweight script checks git repos hourly
- Computes new hashes if commits changed
- Updates org-mode files automatically
- Triggers rebuild if hash changes
- Fail-safe: keeps old hash if fetch/build fails
- You review and approve via git commits

* Security Model

** Defense in Depth
1. Immutable root (can't be modified at runtime)
2. All workloads containerized (isolation)
3. K3s network policies (segmentation)  
4. Headscale mesh (encrypted networking)
5. Passbolt (centralized secret management)
6. Read-only /gnu/store bind-mounted to containers

** Zero-Trust Architecture
- Only essential services on root system
- Everything else in containers
- Container compromise doesn't affect root
- Network mesh authenticates all connections
- Secrets never in environment variables

** Perfect for Security Work
Root system hardened and documented:
- Pentesting tools isolated in containers
- Can test exploits safely
- Defense configurations proven secure
- Every security decision explained in documentation

* Philosophy & Principles

** Core Beliefs
1. *Documentation is not optional* - Understanding before execution
2. *Immutability provides security* - Read-only prevents corruption
3. *Atomicity ensures reliability* - Updates succeed or rollback
4. *Bleeding edge without instability* - Git tracking + fail-safes
5. *Knowledge graph over hierarchy* - Concepts connect, not nest
6. *Explain everything from first principles* - Down to assembly level

** Why This Matters
Traditional operating systems:
- Documentation separate from code (gets stale)
- Updates can break systems (no atomicity)
- Configuration scattered (hard to understand)
- Black boxes everywhere (trust without understanding)

ShapelessOS:
- Documentation IS the system definition
- Updates are atomic (safe by design)
- Everything in knowledge graph (navigable)
- Full transparency (understand everything)

Anyone who values understanding over convenience:

- Security professionals needing auditable systems
- Developers wanting reproducible environments
- Teams needing documented infrastructure
- Anyone with curiosity and patience

"I have no enemy, only victims" - Everyone deserves to understand their computer.

* Development Principles
** Throughout All Phases
*** Documentation First
Every feature must be documented before implementation:
1. Write org-mode documentation explaining what and why
2. Include design rationale and alternatives considered
3. Link to related concepts
4. Implement the feature
5. Update documentation with actual implementation details

No exceptions. If there's no documentation, there's no feature.

*** Fail-Safe Design
Every change must be rollback-able:
- Work in overlays/branches
- Test before commit to main system
- Keep old versions accessible
- Document rollback procedures
- Automatic fallback on failure

Users should never be stuck with broken system.

*** Incremental Progress
Small, working increments over big leaps:
- Each milestone builds on previous
- Working system at every stage
- No "rewrite everything" moments
- Gradual evolution, not revolution

Users can adopt at any phase.

*** Community Transparency
Open development, open decisions:
- Public repositories
- Design discussions in open
- Roadmap changes announced
- User feedback incorporated
- Mistakes acknowledged and learned from

Trust through transparency.

* Risk Mitigation

** Technical Risks

*** Risk: Guix integration more complex than expected
*Mitigation:* Start with simple packages, learn incrementally, fall back to pure Nix if needed

*** Risk: Stow doesn't scale to thousands of packages
*Mitigation:* Test early with 100+ packages, have fallback plan (direct Guix)

*** Risk: Immutable root too restrictive
*Mitigation:* Design overlay system from start, allow escape hatches

*** Risk: Automated updates break things
*Mitigation:* Always test in overlay first, manual approval before deploy

** Scope Risks

*** Risk: Documentation too ambitious (millions of lines)
*Mitigation:* Start with core (100 packages), expand gradually, accept some packages are lightly documented

*** Risk: Feature creep
*Mitigation:* Strict scope per phase, defer non-critical features, focus on 1.0 first

*** Risk: Trying to support all use cases
*Mitigation:* Define clear target audience, specialize for those needs, others can fork

** Community Risks

*** Risk: No adoption
*Mitigation:* Document for yourself first, adoption is bonus not requirement

*** Risk: Negative feedback
*Mitigation:* Expect criticism, learn from valid points, ignore trolls, stay focused on vision

*** Risk: Contributor conflicts
*Mitigation:* Clear contribution guidelines, code of conduct, benevolent maintainership

* Conclusion

ShapelessOS is an ambitious project, but with clear phases and milestones, it's achievable.

**The key principles:**
- Documentation before code
- Small working increments
- Fail-safe at every step
- Transparency and community

**The vision:**
An operating system where understanding is paramount, documentation is inseparable from the system, and users are empowered to know their computers completely.

**The path:**
12 months to 1.0, steady progress, learning and documenting along the way.

**The destination:**
A reference implementation proving that operating systems CAN be fully documented, that understanding DOESN'T require sacrificing practicality, and that knowledge graphs ARE a viable system architecture.
