#+TITLE: ShapelessOS
An Operating System That Doesn't Exist Without Its Documentation

* Introduction

ShapelessOS is a Linux-based operating system built on one principle:
*documentation and code are inseparable*.

Built on Linux From Scratch methodology and powered by GNU Guix,
this system embodies a simple truth:

#+BEGIN_QUOTE
*If there is no documentation, there is no code.*
#+END_QUOTE

Every package, every configuration, every decision exists
as extensively documented org-mode files.
The system doesn't just have documentation —
the documentation *is* the system definition.

"I have no enemy, only victims."

* Architecture Overview

** Layer 0: Immutable Root (Core)
#+BEGIN_SRC

Headscale client (VPN mesh networking)
#+END_SRC

All installed to /gnu/store, frozen read-only after build.

** Layer 1: Infrastructure Services
Running on root system (not containerized):
- Headscale server (mesh network control plane)
- Passbolt server (password/secret management)
- Atuin server (shell history sync)
- K3s control plane — orchestrates all system-declared Docker containers

** Layer 2: Workload Containers
All user applications run containerized, managed by K3s:
- Web development environments
- Pentesting tools (offense and defense)
- Databases and services
- Custom applications

Each container based on the same ShapelessOS root image.

* What Makes ShapelessOS, SHAPELESSOS

** Documentation as Source Code
Your system definition lives in org-mode files:
- Full explanations of what each component does
- Why it exists and what problems it solves
- How it works, down to assembly/machine code level
- Historical context and design evolution
- Source code analysis even for binary packages

** Knowledge Graph Architecture
Files are interconnected nodes, not hierarchical directories:
- Navigate concepts via org-roam backlinks
- Tag and categorize with supertags
- Traverse dependency chains
- Understand relationships between components
- One file per concept, many links to it

** Immutable/Atomic Foundation
- Read-only root filesystem (SquashFS + OSTree + OverlayFS)
- Locked after build — no runtime modifications
- Updates create new system images
- Instant rollback to any previous state
- Zero-trust model: containers for everything except core services

** Bleeding-Edge via Git
- No version numbers, track git repositories directly
- Always up-to-date with upstream
- MAAK monitors local =sources/= clones for new commits and notifies you
- You review, approve, and trigger rebuilds manually
- Fail-safe: nothing updates automatically, you are always in control

** Hybrid Reproducibility
- Guix builds packages reproducibly to =/gnu/store=
- Containers inherit from immutable base
- Best of both worlds: Guix power + full source sovereignty

* Core Technologies

** Guix Package Management
- Reproducible builds from source
- Functional package management
- Rollback capability
- Full Guix DSL — the stack from top to bottom

** MAAK — Automation Backbone
MAAK is a Guile Scheme command runner and project automator.
It is ShapelessOS's automation layer — every system task
is a Scheme function you own, read, and extend.

MAAK owns the following responsibilities:

*** System Build
=maak reconfigure= — runs =guix system reconfigure= against =guixos.scm=

*** Database Population
=maak populate-database= — reads the local Guix source clone
and extracts definitions into =definitions-database/=.
This runs automatically as part of every system build.
You never populate the database manually.

*** Source Monitoring
=maak check-updates= — scans all =sources/= clones,
detects new commits upstream, and notifies you.
Nothing updates automatically. You approve every change.

*** Package Builds
=maak build <package>= — compiles a package from its local =sources/= clone,
using its self-contained =.scm= module definition.

*** ISO Build
=maak build-iso= — produces a bootable ShapelessOS ISO
declared directly in the system configuration.
If your machine dies and you have the repo,
this is how you get back to a running system.

** Emacs Integration
- Custom Emacs package (ldos-mode.el) included

- Automatic hash computation (C-c C-h)
- Build packages from org blocks (C-c C-b)
- Navigate knowledge graph via org-roam
- Your custom Emacs distro comes preconfigured

** SquashFS + OSTree + OverlayFS
- SquashFS: Compressed immutable base
- OSTree: Version management and atomic updates
- OverlayFS: Configuration overlays
- Combined: Secure, efficient, rollback-capable

* Package Sovereignty

ShapelessOS has zero runtime external dependencies.
No channels pointing to remote servers.
No network fetches during build.
No substitute servers.
GuixOS servers going dark is completely irrelevant.

** How It Works

The Guix and nonguix repositories are cloned once into =~/.config/guix/=
and treated as a read-only reference database — never imported,
never referenced at runtime. From them, MAAK extracts raw definitions
into =definitions-database/= at every system build.

When you write a package module, you open =definitions-database/=,
copy the definitions you need directly into the file,
and point the source at your local =sources/= clone.
The file becomes a closed world. No imports from anywhere outside itself.

Delete the file → the package never existed on your machine.

** Self-Contained Modules

Each =.scm= package file:
- Copies definitions directly from =definitions-database/=
- Points source at =file://= URI inside =sources/=
- Has zero =use-module= references outside itself
- Is the single source of truth for that package

This means:
- You own the source code
- You own the build definitions
- You can patch anything directly in =sources/=
- Rollback is git history — no extra mechanism needed

* Module Architecture — Loaders

ShapelessOS uses a modular loader architecture
to organize and compose system packages and services.

Every concern lives in its own focused =.scm= file.
Loaders aggregate those files into named groups.
The system declaration consumes loaders — never individual files.

** Why This Matters

- Add a package: create one =.scm= file, add it to the relevant loader
- Remove a package: delete the =.scm= file and its =sources/= clone
- Nothing else in the system needs to change
- The loader is the only coupling point between a module and the system

* Package Management Philosophy

** Three Categories of Packages

*** Stable from Guix Upstream

- glibc, gcc, binutils

*** Bleeding Edge from Git

*** Custom ShapelessOS Packages
Our own code and configurations:
- ldos-tools, ldos-init
- Custom system utilities
- Specialization-specific software

Fully owned — source in =sources/=, definitions written from scratch.

* Security Model

** Defense in Depth
1. Immutable root (can't be modified at runtime)
2. All workloads containerized (isolation)
3. K3s network policies (segmentation)
4. Headscale mesh (encrypted networking)
5. Passbolt (centralized secret management)
6. Read-only =/gnu/store= bind-mounted to containers

** Zero-Trust Architecture
- Only essential services on root system
- Everything else in containers
- Container compromise doesn't affect root
- Network mesh authenticates all connections
- Secrets never in environment variables

** Perfect for Security Work
Root system hardened and documented:
- Pentesting tools isolated in containers
- Can test exploits safely
- Defense configurations proven secure
- Every security decision explained in documentation

* Philosophy & Principles

** Core Beliefs
1. *Documentation is not optional* — Understanding before execution
2. *Sovereignty is not optional* — Own everything from source to build
3. *Immutability provides security* — Read-only prevents corruption
4. *Atomicity ensures reliability* — Updates succeed or rollback
5. *Bleeding edge without instability* — Git tracking + manual approval
6. *Knowledge graph over hierarchy* — Concepts connect, not nest
7. *Explain everything from first principles* — Down to assembly level

** Why This Matters
Traditional operating systems:
- Documentation separate from code (gets stale)
- Updates can break systems (no atomicity)
- Configuration scattered (hard to understand)
- External dependencies you don't control
- Black boxes everywhere (trust without understanding)

ShapelessOS:
- Documentation IS the system definition
- Updates are atomic (safe by design)
- Everything in knowledge graph (navigable)
- Zero runtime external dependencies
- Full transparency — understand everything

* Development Principles

** Throughout All Phases

*** Documentation First
Every feature must be documented before implementation:
1. Write org-mode documentation explaining what and why
2. Include design rationale and alternatives considered
3. Link to related concepts
4. Implement the feature
5. Update documentation with actual implementation details

No exceptions. If there's no documentation, there's no feature.

*** Fail-Safe Design
Every change must be rollback-able:
- Work in overlays/branches
- Test before commit to main system
- Keep old versions accessible via git history
- Document rollback procedures
- Automatic fallback on failure

Users should never be stuck with a broken system.

*** Incremental Progress
Small, working increments over big leaps:
- Each milestone builds on previous
- Working system at every stage
- No "rewrite everything" moments
- Gradual evolution, not revolution

*** Community Transparency
Open development, open decisions:
- Public repositories
- Design discussions in open
- Roadmap changes announced
- User feedback incorporated
- Mistakes acknowledged and learned from

Trust through transparency.

* Risk Mitigation

** Technical Risks

*** Risk: Guix integration more complex than expected
*Mitigation:* Start with simple packages, learn incrementally

*** Risk: Immutable root too restrictive
*Mitigation:* Design overlay system from start, allow escape hatches

*** Risk: Source monitoring misses breaking changes
*Mitigation:* MAAK notifies, human approves — no silent updates ever

** Scope Risks

*** Risk: Documentation too ambitious
*Mitigation:* Start with core packages, expand gradually,
accept some packages are lightly documented initially

*** Risk: Feature creep
*Mitigation:* Strict scope per phase, defer non-critical features,
focus on 1.0 first

** Community Risks

*** Risk: No adoption
*Mitigation:* Document for yourself first, adoption is bonus not requirement

*** Risk: Negative feedback
*Mitigation:* Expect criticism, learn from valid points,
ignore trolls, stay focused on vision

*** Risk: Contributor conflicts
*Mitigation:* Clear contribution guidelines, code of conduct,
benevolent maintainership
