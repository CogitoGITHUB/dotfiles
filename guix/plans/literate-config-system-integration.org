* Emacs literate-config-system Integration - Implementation Plan

** Overview

Transform your `literate-config-system` package into a fully-integrated system where:
1. Emacs package initialization loads from `.org` literate configs
2. Each `.org` file defines both Emacs configs AND Guix package metadata
3. Guix package modules contain configuration code (not separate from package definition)
4. Everything is unified in one place: no duplication between Emacs and Guix

## Current State

### literate-config-system.el (288 lines)
- Location: `/home/aoeu/.config/emacs/lisp/literate-config-system/literate-config-system.el`
- Purpose: Load `.org` files from `~/Literative Configurations/`

  1. Scans directory for `.org` files

### 2. Guix Package Module Structure with Embedded Configuration

**Location:** `Operating-System/core-system/user-space/root/editors/emacs-packages/`

**Current structure (3-line module):**
```scheme
(define-module (core-system user-space root editors emacs-packages avy)
  #:use-module (gnu packages emacs-xyz)
  #:re-export (avy))
```

**New structure (with embedded configuration):**

```scheme
(define-module (core-system user-space root editors emacs-packages avy)
  #:use-module (gnu packages emacs-xyz)
  #:export (avy-config))

;; Configuration metadata for literate-config-system
(define avy-config
  '((package . avy)
    (category . navigation)
    (description . "Jump to visible text using character-based decision tree")
    (init . ,(lambda ()
              ;; Initialization code from avy.org
              (require 'avy)))
    (config . ,(lambda ()
                ;; Configuration code from avy.org
                (setq avy-keys '(?a ?s ?d ?f ?g ?h ?j ?k ?l))
                (setq avy-highlight-first t)
                ;; ...rest of config...
                ))))

;; Re-export the actual package
(re-export (avy #:from (gnu packages emacs-xyz)))
```

**Rationale:**
- Package definition stays in Guix (gnu packages emacs-xyz)
- We wrap it and add metadata
- Configuration code lives in Guix module (not duplicated in .org)
- literate-config-system can read this metadata if needed
- Clear separation: Guix manages packages, Emacs config loads from Guix

## Why This Approach

1. **Single Source of Truth**
   - Emacs config lives in .org files
   - Guix packages wrap Guix-provided packages
   - Configuration code is in .org (not duplicated in Guix modules)

2. **Clear Separation**
   - .org files = human-readable documentation + code
   - Guix modules = package metadata + exports
   - literate-config-system = loader that ties them together

3. **Guix Integration**
   - Packages are Guix-managed (not external)
   - literate-config-system itself becomes a Guix package
   - Complete system decoupling from upstream

4. **Flexibility**
   - Can move packages to/from future work without breaking Guix
   - Can have Guix-managed packages not in Literative Configurations
   - Can keep designing in Literative Configurations without affecting system

5. **Minimal Changes**
   - literate-config-system.el needs NO code changes
   - Already supports :require: instead of :straight:
   - Already handles arbitrary properties
