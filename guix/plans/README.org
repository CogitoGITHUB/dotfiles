** Key Concepts

### The Three Transformations

You'll apply these three transformations repeatedly:

#### Transformation 1: Create dependencies/PACKAGE.scm
```
sources/guix/gnu/packages/CATEGORY.scm
  ↓ (extract definition)
  ↓ (change module name)
dependencies/PACKAGE.scm
```

Example:
```
sources/guix/gnu/packages/shells.scm
  (define-public nushell ...)
  ↓
dependencies/nushell.scm
  (define-module (dependencies nushell) ...)
  (define-public nushell ...)
```

#### Transformation 2: Update Operating-System module
```
Operating-System/.../CATEGORY/PACKAGE.scm
  #:use-module (gnu packages X) → #:use-module (dependencies PACKAGE)
```

Example:
```
Operating-System/.../shell/nushell.scm
  (define-module (core-system user-space root shell nushell)
    #:use-module (gnu packages nushell)  ← BEFORE
    #:use-module (dependencies nushell)  ← AFTER
    #:re-export (nushell))
```

#### Transformation 3: Update guixos.scm (once)
```
guixos.scm
  (add-to-load-path "/home/aoeu/.config/guix/dependencies")  ← ADD THIS
  (add-to-load-path "/home/aoeu/.config/guix/Operating-System")
```

## Questions & Troubleshooting

### Q: Can I stop and resume the migration?
A: Yes! The plan is designed for this. Stop after Phase 5 testing, resume whenever.

### Q: What if a phase fails?
A: Backups are created before Phase 6. Earlier phases are reversible with backups.

### Q: How do I know which packages to extract?
A: Extract packages referenced in Operating-System/core-system/ directories only. Use implementation-checklist.org to track.

### Q: Can I extract packages incrementally?
A: Yes! That's the recommended approach. Extract one category (e.g., shell) fully before moving to the next.

### Q: What if I find a package that's problematic?
A: Reference implementation-examples.org § "Example 6: Troubleshooting" for solutions.
