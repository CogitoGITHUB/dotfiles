* Troubleshooting

** Golden rule: REPL first, never reconfigure to debug a module
Every error below can be caught in REPL. See guix-repl.org.

** Error: "<symbol>: unbound variable"

Cause: symbol used but not exported/imported somewhere in the chain.
Test each layer bottom-up in REPL:

#+BEGIN_SRC scheme
(use-modules (core-system user-space root <category> <name>))    ; submodule
(use-modules (core-system user-space root loaders <category>))   ; loader
(use-modules (core-system user-space root root))                  ; root
#+END_SRC

Fix checklist:
- Submodule has #:re-export (<symbol>) or #:export (<symbol>)?
- Loader has #:use-module for that submodule?
- Loader has #:re-export (<symbol>)?
- Loader has symbol in root-*-packages list?
- root.scm has #:use-module for that loader?

** Error: "<symbol>: unbound variable" where symbol is a gnu package (e.g. shadow)

Cause: Missing #:use-module (gnu packages <module>) in submodule or loader.

Find correct module in REPL:
#+BEGIN_SRC scheme
,apropos shadow
#+END_SRC

Or from shell:
#+BEGIN_SRC sh
guix search shadow
# check: location: gnu/packages/<module>.scm
#+END_SRC

Add the missing import to your submodule.

** Error: "no code for module (core-system user-space root <category> <name>)"

Cause: Wrong module path OR file missing.

Check:
1. File exists: ls Operating-System/core-system/user-space/root/<category>/<name>.scm
2. define-module in file matches exactly: (core-system user-space root <category> <name>)
3. REPL launched with correct flag: -L /home/aoeu/.config/guix/Operating-System

** Error: "invalid-base32-character" or hash format errors

Cause: Wrong hash tool used.

#+BEGIN_SRC sh
guix download <url>
# copy result directly as: (base32 "0abc...")
#+END_SRC
Never use sha256sum or guix hash.

** Error: "wrong type argument" in operating-system

Cause: A root-*-packages list contains a non-package object, or service list contains a non-service.

#+BEGIN_SRC scheme
(use-modules (core-system user-space root loaders <category>))
root-<category>-packages  ; inspect the actual value
#+END_SRC

** System won't boot

#+BEGIN_SRC sh
# At GRUB: select previous generation entry
sudo guix system roll-back
sudo guix system switch-generation N
#+END_SRC

** Service not starting

#+BEGIN_SRC sh
sudo herd status <service-name>
sudo herd log <service-name>
sudo herd start <service-name>
#+END_SRC

** Error: "error while loading shared libraries: libstdc++.so.6" (prebuilt binary)

Cause: Prebuilt binary needs libstdc++ at runtime but the rpath doesn't include gcc's lib output.

Fix: Add gcc (lib output) to inputs and set rpath with patchelf:
#+BEGIN_SRC scheme
(inputs (list bash tar zstd patchelf glibc `(,gcc "lib")))
;; In builder:
(let* ((glibc (assoc-ref %build-inputs "glibc"))
       (gcc (assoc-ref %build-inputs "gcc"))  ; this is the "lib" output
       (interp (string-append glibc "/lib/ld-linux-x86-64.so.2"))
       (libstdc++ (string-append gcc "/lib")))
  (invoke patchelf "--set-interpreter" interp "--set-rpath"
          (string-append glibc "/lib:" libstdc++) binary))
#+END_SRC

Note: Guix splits gcc into multiple outputs. The "lib" output contains libstdc++.so.6.
When using `(,gcc "lib")` as an input, it's still named "gcc" in %build-inputs but
points to the lib output directory (e.g. gcc-14.3.0-lib).

** Error: "Cannot open: No such file or directory" during tar extraction

Cause: The output directory doesn't exist before tar tries to extract to it.

Fix: Add `(mkdir-p out)` before the tar extraction command.

** Error: "rename-file: No such file or directory" with prebuilt tarball

Cause: Extracting to wrong directory. If archive has `bin/ollama` inside it and you
use `tar xf - -C out/bin`, it extracts to `out/bin/bin/ollama`.

Fix: Extract to `out/` instead of `out/bin/`. The archive's internal directory
structure will resolve correctly: `bin/ollama` -> `out/bin/ollama`.

** Post-reconfigure: always test package binaries

A successful reconfigure does NOT mean packages work. Always test each new binary:

#+BEGIN_SRC sh
# After reconfigure, test every new package:
<package> --version 2>&1 || <package> -v 2>&1 || <package> -V 2>&1 || <package> 2>&1
#+END_SRC

Common patterns:
- `--version` works for most packages (rmpc, pipewire, pulseaudio, amixer)
- `-v` or `-V` for some (cava uses `-v`)
- Just running the binary — some exit with error but that's ok if it runs
- For services: `sudo herd status <name>` and check `/var/log/messages`

Example for audio packages:
#+BEGIN_SRC sh
rmpc --help 2>&1; cava -v 2>&1; pipewire --version 2>&1; \
pulseaudio --version 2>&1; amixer --version 2>&1
#+END_SRC

If any binary returns "command not found" — the package wasn't built into the profile.
Check: `guix build <package>` and `guix package -I <package>`.

** Emacs package fails to byte-compile with "Cannot open load file"

Cause: The package has `(require 'some-dep)` at the top level but the dependency
isn't available during the build phase.

Fix: Add the dependency as BOTH `inputs` AND `propagated-inputs`:
#+BEGIN_SRC scheme
(inputs (list emacs-<dep>))
(propagated-inputs (list emacs-<dep>))
#+END_SRC

`inputs` makes it available at compile time. `propagated-inputs` makes it
available at runtime. You need BOTH for top-level requires.

** Hash mismatch on git-fetch packages

Cause: `guix hash --recursive /path/to/checkout` includes `.git/` directory,
but Guix's `git-fetch` excludes it. The hashes will never match.

Fix: Use the actual hash from the failed build log:
#+BEGIN_SRC sh
zcat /var/log/guix/drvs/<XX>/<hash>.drv.gz | grep "actual hash"
#+END_SRC

The actual hash always starts with `0` or `1` (base32).

** Python package sanity-check fails due to version mismatch

Cause: The package requires a newer version of a dependency than Guix provides.
The pyproject-build-system's sanity-check phase validates version constraints.

Fix: Delete the sanity-check phase:
#+BEGIN_SRC scheme
(arguments
 '(#:tests? #f
   #:phases
   (modify-phases %standard-phases
     (delete 'sanity-check))))
#+END_SRC

** Go package build fails — "go.mod needs updating" or "go install requires a version"

Cause 1: The `go.mod` specifies a Go version newer than Guix provides.
Fix: Patch go.mod in a source snippet:
#+BEGIN_SRC scheme
(modules '((guix build utils)))
(snippet #~(begin
             (substitute* "go.mod"
               (("go 1.25.5") "go 1.24"))))
#+END_SRC

Cause 2: `#:import-path` points to `cmd/<name>/` instead of the module root.
The go.mod is at the root — `#:import-path` must match.
Fix: Use root import path, override build phase to install cmd/:
#+BEGIN_SRC scheme
#:import-path "github.com/user/repo"   ; NOT github.com/user/repo/cmd/name
#:phases #~(modify-phases %standard-phases
    (replace 'build
      (lambda* (#:key import-path #:allow-other-keys)
        (invoke "go" "install" "-ldflags=-s -w" "-trimpath"
                (string-append import-path "/cmd/name")))))
#+END_SRC

Cause 3: Go dependency version mismatch between go.mod and what's in Guix.
Fix: Check versions with `guix package --show=go-github-com-<name>` and compare
to go.mod. If versions don't match, you may need to package the correct version.

** npm FOD build fails — "EACCES: permission denied, mkdir '/homeless-shelter'"

Cause: npm tries to write to $HOME/.npm which doesn't exist in the build sandbox.
Fix: Set HOME and npm config vars to the output directory:
#+BEGIN_SRC scheme
(setenv "HOME" out)
(setenv "npm_config_cache" tmp)
(setenv "npm_config_tmp" tmp)
#+END_SRC

** npm FOD build fails — "tar: gzip: Cannot exec"

Cause: tar needs gzip for .tgz files but gzip isn't in PATH.
Fix: Use `--use-compress-program` with the full gzip path:
#+BEGIN_SRC scheme
(invoke tar "--use-compress-program" gzip "-xf" src "-C" work)
#+END_SRC
And add both tar and gzip to inputs.

** n8n (or similar) runs but --version shows nothing

Cause: The wrapper script points to `dist/index.js` instead of `bin/<name>`.
The bin script has version checks, config loading, and module imports.
Fix: Point wrapper to the bin entry point:
#+BEGIN_SRC scheme
(format p "#!/bin/sh\nexec ~a ~a/bin/<name> \"$@\"\n" node lib)
#+END_SRC


** WiFi device unavailable: cfg80211 regulatory.db signature error

Error in dmesg:
: cfg80211: loaded regulatory.db is malformed or signature is missing/invalid

Cause: Linux kernel's cfg80211 requires a signed regulatory database. The signature file
(regulatory.db.p7s) must match the database exactly.

Fix: Disable signature checks in kernel config via nonguix:

In kernel-space/linux.scm:
#+BEGIN_SRC scheme
(define-public kernel
  (package
    (inherit (corrupt-linux linux-libre-6.18
                          #:configs '()))
    (name "linux-with-regdb-fix")
    (source
     (origin
       (inherit (package-source linux-libre-6.18))
       (snippet
        #~(begin
            (when (file-exists? "arch/x86/configs/guix_defconfig")
              (let ((defconfig "arch/x86/configs/guix_defconfig"))
                (substitute* defconfig
                  (("CONFIG_CFG80211_REQUIRE_SIGNED_REGDB=y")
                   "CONFIG_CFG80211_REQUIRE_SIGNED_REGDB=n")
                  (("CONFIG_CFG80211_USE_KERNEL_REGDB_KEYS=y")
                   "CONFIG_CFG80211_USE_KERNEL_REGDB_KEYS=n"))
                (format #t "Patched ~a~%" defconfig)))))))))
#+END_SRC

This patches the guix_defconfig to disable signature verification before the kernel
builds. The kernel will then accept any regulatory.db without signature validation.

** Linux-libre blocks non-free firmware

With linux-libre kernel:
: Missing Free firmware (non-Free firmware loading is disabled)
: Direct firmware load for rtw88_8821ce failed

Cause: linux-libre refuses to load proprietary firmware blobs.

Fix: Use nonguix's corrupt-linux instead, which includes non-free firmware.

Note: Realtek RTL8821CE and similar chips REQUIRE proprietary firmware to function.
