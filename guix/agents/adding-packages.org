* Adding Packages and Services

** Decision Tree — before writing any code

Is the package already in Guix proper?
  YES -> Template A (thin submodule, just #:re-export)
         confirm: guix search <name>
  NO  -> search web for existing Guix channel recipe
           FOUND     -> copy define-public from channel source -> Template B
           NOT FOUND -> write from scratch                     -> Template B
                        get hash: guix download <url>

** No external channels — ever
Never add to channels.scm.
Copy package definitions directly into your submodule. Self-contained always.

** Step-by-step: adding a new package

*** Step 1 — Pick category
See module-reference.org for all categories.
No fitting category -> create new dir + loader (Template C + Template D).

*** Step 2 — Create submodule file
File:   Operating-System/core-system/user-space/root/<category>/<name>.scm
Module: (core-system user-space root <category> <name>)
Use Template A or B from module-templates.org.

*** Step 3 — Validate submodule in REPL
#+BEGIN_SRC scheme
(use-modules (core-system user-space root <category> <name>))
<name>   ; must return package object, not an error
#+END_SRC
Do not proceed if this fails.

*** Step 4 — Update the loader
In loaders/<category>.scm:
1. Add #:use-module (core-system user-space root <category> <name>)
2. Add symbol to #:re-export (...)
3. Add symbol to root-<category>-packages list
Validate loader in REPL before continuing.

*** Step 5 — Update root.scm (new loader only)
1. Add #:use-module (core-system user-space root loaders <category>)
2. Append root-<category>-packages to root-system-packages
3. If services: append root-<category>-services to root-system-services

*** Step 6 — Validate root in REPL
#+BEGIN_SRC scheme
(use-modules (core-system user-space root root))
root-system-packages  ; must load clean
#+END_SRC

*** Step 7 — Reconfigure
#+BEGIN_SRC sh
sudo guix system reconfigure /home/aoeu/.config/guix/guixos.scm
#+END_SRC

*** Step 8 — Test each package binary works
After reconfigure, verify every new package actually runs — not just that the system reconfigures:
#+BEGIN_SRC sh
# Test each binary: --version, -v, -V, or just run it
<package> --version 2>&1
<package> -v 2>&1
<package> -V 2>&1
<package> 2>&1  # some packages exit with error but that's ok if binary runs
#+END_SRC
For services:
#+BEGIN_SRC sh
sudo herd status <service-name>  # should show running
sudo grep -i <service-name> /var/log/messages | tail -5  # check for errors
#+END_SRC
If a binary is "command not found" or a service logs errors — FIX IT. Do not skip this step.

** Getting the correct hash
#+BEGIN_SRC sh
guix download https://example.com/package-1.0.tar.gz
# use result as: (base32 "0abc...")
#+END_SRC
Never use sha256sum or guix hash — wrong format.

** npm packages

For packages installed via npm (Node.js):

1. Find version:
   #+BEGIN_SRC sh
   npm show <package> version   # or check npmjs.com
   #+END_SRC

2. Construct tarball URL:
   #+BEGIN_SRC
   https://registry.npmjs.org/<package>/-/<package>-<version>.tgz
   #+END_SRC
   Example for openclaw v2026.3.28:
   https://registry.npmjs.org/openclaw/-/openclaw-2026.3.28.tgz

3. Get hash:
   #+BEGIN_SRC sh
   guix download https://registry.npmjs.org/openclaw/-/openclaw-2026.3.28.tgz
   #+END_SRC

4. Use node-build-system in Template B.

** npm packages with dependencies

When an npm package has runtime dependencies (most do), the npm tarball from
the registry does NOT include node_modules. The build sandbox blocks network
access, so `npm install` fails. Two approaches:

*** Approach 1: Pre-built tarball (simplest for personal use)

1. Install the package locally: `npm install -g <package>`
2. Find the global node_modules directory: `npm prefix -g`
3. Create a tarball of the entire node_modules:
   #+BEGIN_SRC sh
   cd $(npm prefix -g)/lib
   tar czf <package>-<version>-with-deps.tar.gz node_modules
   #+END_SRC
4. Place the tarball in your config tree (e.g., `ai/sources/<package>-<version>.tar.gz`)
5. Use `local-file` as source with `trivial-build-system`:
   #+BEGIN_SRC scheme
   (source (local-file "sources/<package>-<version>.tar.gz"))
   ;; Builder: extract to out/, create wrapper script pointing to entry point
   #+END_SRC
6. Set `propagated-inputs` to `(list node)` so node is available at runtime

*** Approach 2: guix import npm-binary (for full Guix integration)

The `guix import npm-binary <package>` command generates a package expression
that lists all dependencies as individual `node-*` package inputs. This requires
all dependencies to exist in Guix — if any are missing, you must package them
too. Works well for small packages with few dependencies, impractical for large
ones like openclaw (~90 deps).

*** Key insight

The npm registry tarball (`registry.npmjs.org/<pkg>/-/<pkg>-<version>.tgz`)
contains only source code, NOT node_modules. Any approach using this as source
must either: (a) run npm install during build (needs network, not available
in sandbox), or (b) bundle node_modules separately (approach 1 above).

** Adding a service
#+BEGIN_SRC scheme
;; In the loader:
(define-public root-<category>-services
  (list (service your-service-type)))

;; In root.scm:
(define-public root-system-services
  (append ...
          root-<category>-services
          ...))
#+END_SRC
